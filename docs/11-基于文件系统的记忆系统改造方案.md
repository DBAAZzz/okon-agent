# 基于文件系统的记忆系统改造方案

## 1. 背景与目标

当前项目的记忆系统基于 Session + Qdrant 向量数据库：

- Session 历史 → PostgreSQL（Message 表）
- 会话记忆 → Qdrant `memories` 集合（session 级，全量写入）
- 知识库 → Qdrant `kb_*` + PostgreSQL（Document/SourceFile）
- 上下文压缩 → LLM 生成 CompactionSummary 存 PG

存在的问题：

1. **全量写入**：每轮对话都存入 Qdrant，大量无价值内容污染记忆
2. **缺乏结构化分层**：所有记忆平铺存储，无优先级、无生命周期管理
3. **不可审查**：记忆存在 Qdrant 中，人类无法直接阅读和管理
4. **无跨 Session 记忆**：记忆按 sessionId 隔离，无法形成长期用户画像

改造目标：文件系统做 Agent 认知交互层，引入 LLM 事实提取替代全量写入。

---

## 2. 格式选择：Markdown 优先，JSONL 做补充

### 2.1 核心原则

文件系统作为 Agent 记忆载体，核心价值是 **LLM 能自然读写、人能直接理解**。Markdown 完美契合这两点，应该是主力格式。JSONL 只在需要程序化批量处理的场景下才引入。

| 维度 | Markdown | JSONL |
| ---- | -------- | ----- |
| LLM 读取 | 直接注入 prompt，零转换 | 需要 parse 再拼文本 |
| LLM 写入 | 生成自然文本，几乎不会出格式错误 | 需要严格 JSON，容易出错 |
| 人类阅读 | 直接打开就能看懂 | 需要工具或肉眼解析 |
| 程序解析 | 需要约定格式 | 天然结构化 |
| Git diff | 清晰可读 | 可读但密度高 |

**结论：记忆文件用 Markdown，只在未来需要批量程序化处理时才引入 JSONL。**

### 2.2 存储结构

```text
memory/{botId}/
├── memories.md          # 所有 active 记忆（主文件，LLM 直接读写）
└── archive.md           # 被取代/过期的记忆（审计用，平时不读）
```

就两个文件。不做多层目录，不做 .abstract 索引。百条级记忆一个 Markdown 文件完全够用。

### 2.3 memories.md 格式

```markdown
# Bot 记忆

## preference

- [P0] 项目统一使用 pnpm 作为包管理器，不使用 npm <!-- id:a1b2c3 session:42 updated:2026-02-20 -->
- [P0] 回复风格要求：专业、简洁、少废话 <!-- id:d4e5f6 session:15 updated:2026-01-10 -->

## fact

- [P1] 部署环境为 Ubuntu 24.04 <!-- id:g7h8i9 session:88 updated:2026-02-25 -->
- [P1] 数据库使用 PostgreSQL 16 + Prisma ORM <!-- id:j0k1l2 session:30 updated:2026-02-01 -->

## lesson

- [P1] Qdrant 连接超时的解决方案：连接池需设置 keepAliveTimeoutMs: 30000 <!-- id:m3n4o5 session:65 updated:2026-02-15 -->

## entity

## intent

- [P2] 计划将记忆系统从向量数据库迁移到文件系统 <!-- id:p6q7r8 session:90 updated:2026-02-26 -->
```

**格式约定：**

- 按 category 分组（二级标题）
- 每条记忆是一个列表项：`- [优先级] 内容陈述`
- HTML 注释 `<!-- -->` 存放元数据（id、来源 session、更新时间），LLM 读取时自然忽略，人类审查时可见
- 内容必须是自包含陈述句，脱离上下文也能理解

### 2.4 archive.md 格式

```markdown
# 归档记忆

- ~~[P1] 部署环境为 Ubuntu 22.04~~ → 被 g7h8i9 取代 <!-- id:old123 session:30 archived:2026-02-25 -->
```

被取代的记忆移到这里，删除线标记，注明被谁取代。平时不加载，只在审计时查看。

## 3. 核心机制：LLM 事实提取

### 3.1 设计思路

一次 LLM 调用同时完成事实提取和记忆决策。**提取不出 = 不记**，空数组就是"不值得记"。

将已有记忆直接喂给 LLM（就是 memories.md 的内容），让模型自己判断是新增、更新还是什么都不做。LLM 能理解语义等价关系（如"Ubuntu 22.04"和"部署环境是 Ubuntu"是同一事实），比代码侧做 tags 交集匹配更准确。

### 3.2 提取 Prompt

```text
你是一个记忆管理器。分析以下对话，判断是否需要更新用户的长期记忆。

## 提取规则

提取以下类型：
- preference: 用户偏好、习惯、风格要求
- fact: 项目事实、技术选型、架构决策、配置信息
- entity: 人物、项目、工具之间的关系
- lesson: 经验教训、踩坑记录、解决方案
- intent: 长期目标、计划、待办

不要提取：
- 闲聊、寒暄、礼貌用语
- 一次性查询的答案本身（如翻译结果、天气）
- 工具执行的原始输出
- 过程性对话（"好的"、"继续"、"帮我看看"）
- 已经被后续对话纠正/推翻的信息
- 密码、密钥、token 等敏感信息

## 当前记忆

{existingMemories}

## 本轮对话

用户：{userMessage}
助手：{assistantMessage}

## 输出格式

返回 JSON 数组。如果没有值得记忆的内容，返回 []。

新增事实：
{ "action": "create", "content": "陈述句", "category": "preference|fact|entity|lesson|intent", "priority": "P0|P1|P2" }

更新已有事实（内容变化、被推翻）：
{ "action": "update", "targetId": "要更新的记忆id", "content": "新的陈述", "priority": "P0|P1|P2" }

删除记忆（用户明确要求忘掉）：
{ "action": "delete", "targetId": "要删除的记忆id" }

## 关键约束
- content 必须是自包含陈述句，脱离上下文也能理解
- 不要凑数，返回 [] 完全正常
- 优先提取用户明确表达的，而非推测的
- 如果用户说"记住xxx"或"以后都xxx"，priority 至少 P1
- 如果新事实与已有记忆矛盾，用 update 更新旧记忆，不要同时存两个版本
- 每条记忆的 id 存在 HTML 注释 <!-- id:xxx --> 中，update/delete 时请从中读取对应 id
```

### 3.3 提取效果示例

**对话 1 — 一次性操作，不记：**

> 用户："帮我把这个函数改成 async 的"

提取结果：`[]`

**对话 2 — 多条新事实：**

> 用户："我们项目统一用 pnpm，不要用 npm。另外部署环境是 Ubuntu 22.04"

```json
[
  { "action": "create", "content": "项目统一使用 pnpm 作为包管理器，不使用 npm", "category": "preference", "priority": "P0" },
  { "action": "create", "content": "部署环境为 Ubuntu 22.04", "category": "fact", "priority": "P1" }
]
```

**对话 3 — 事实变更：**

> 用户："我们升级到 Ubuntu 24.04 了"

```json
[
  { "action": "update", "targetId": "g7h8i9", "content": "部署环境为 Ubuntu 24.04", "priority": "P1" }
]
```

**对话 4 — 用户要求忘掉：**

> 用户："忘掉之前记的那个 RPC 地址"

```json
[
  { "action": "delete", "targetId": "x1y2z3" }
]
```

### 3.4 输出校验与容错

```typescript
import { z } from 'zod'

const MemoryActionSchema = z.discriminatedUnion('action', [
  z.object({
    action: z.literal('create'),
    content: z.string().min(5),
    category: z.enum(['preference', 'fact', 'entity', 'lesson', 'intent']),
    priority: z.enum(['P0', 'P1', 'P2']),
  }),
  z.object({
    action: z.literal('update'),
    targetId: z.string(),
    content: z.string().min(5),
    priority: z.enum(['P0', 'P1', 'P2']),
  }),
  z.object({
    action: z.literal('delete'),
    targetId: z.string(),
  }),
])
```

**容错原则：解析失败就不记。** 不重试，不降级。宁可漏记，不可乱记。

```typescript
function parseExtractResult(raw: string): MemoryAction[] {
  try {
    const jsonMatch = raw.match(/\[[\s\S]*\]/)
    if (!jsonMatch) return []
    const parsed = JSON.parse(jsonMatch[0])
    return z.array(MemoryActionSchema).parse(parsed)
  } catch {
    logger.warn('记忆提取结果解析失败，跳过本轮')
    return []
  }
}
```

### 3.5 调用时机与成本

```text
对话结束（异步，不阻塞主响应流）
   ↓
extractMemories(userMsg, assistantMsg, existingMemories)
   ↓
返回 [] → 结束（大部分对话走这条路）
返回 [action, ...] → 执行 create/update/delete → 更新 memories.md
```

- 输入：用户消息 + 助手回复 + 已有记忆（通常 < 3000 tokens）
- 输出：JSON 数组（通常 < 300 tokens）
- 使用 deepseek-chat，单次 < ¥0.002
- 异步执行，不阻塞主响应流

## 4. 文件读写实现

### 4.1 部署约束

**本方案假设单实例 Node.js 进程。** 内存 WriteQueue 在单进程内序列化写操作即可保证一致性。多实例部署时需要引入外部锁（如 Redis），但当前不需要。

### 4.2 写入保障

Markdown 文件的更新是"读 → 改 → 写"模式，用 tmp + rename 保证原子性：

```typescript
async function updateMemoryFile(botId: number, updater: (content: string) => string) {
  const filepath = getMemoryPath(botId)
  const content = await fs.readFile(filepath, 'utf-8').catch(() => INITIAL_TEMPLATE)
  const updated = updater(content)
  await fs.writeFile(filepath + '.tmp', updated)
  await fs.rename(filepath + '.tmp', filepath)  // 原子替换
}
```

Node.js 单线程中，用内存队列序列化写操作，防止并发冲突：

```typescript
class WriteQueue {
  private queue: Promise<void> = Promise.resolve()
  enqueue(fn: () => Promise<void>): Promise<void> {
    this.queue = this.queue.then(fn, fn)
    return this.queue
  }
}
```

### 4.3 Action 执行

```typescript
async function executeActions(botId: number, actions: MemoryAction[], sessionId: number) {
  await writeQueue.enqueue(async () => {
    await updateMemoryFile(botId, (content) => {
      for (const action of actions) {
        switch (action.action) {
          case 'create':
            content = insertMemory(content, action, sessionId)
            break
          case 'update':
            // targetId 不存在 → 降级为 create（LLM 可能编造 id）
            if (hasMemoryId(content, action.targetId)) {
              content = replaceMemory(content, action)
            } else {
              logger.warn('update targetId 不存在，降级为 create', { targetId: action.targetId })
              content = insertMemory(content, { ...action, action: 'create', category: inferCategory(content, action) }, sessionId)
            }
            break
          case 'delete':
            if (hasMemoryId(content, action.targetId)) {
              const { updated, archivedLine } = removeMemory(content, action.targetId)
              content = updated
              // 归档到 archive.md，失败不影响主流程
              appendToArchive(botId, archivedLine).catch(err =>
                logger.warn('archive 写入失败，不影响主流程', err)
              )
            } else {
              logger.warn('delete targetId 不存在，跳过', { targetId: action.targetId })
            }
            break
        }
      }
      return content
    })
  })
}
```

- `insertMemory`：在对应 category 分组下追加一行
- `replaceMemory`：找到 targetId 所在行，原地替换内容（不归档，旧值无保留价值）
- `removeMemory`：从 memories.md 移除该行，返回被移除的行用于归档
- `hasMemoryId`：检查 content 中是否存在 `<!-- id:targetId`，防止 LLM 编造 id

## 5. 记忆检索

### 5.1 加载策略

Agent 运行前读取 memories.md，直接注入系统提示词：

```typescript
async function loadMemories(botId: number): Promise<string> {
  const filepath = getMemoryPath(botId)
  const content = await fs.readFile(filepath, 'utf-8').catch(() => '')
  return content
}
```

**直接读文件、直接注入。** 不需要 parse 成对象再拼回文本，Markdown 本身就是最好的 prompt 格式。

### 5.2 注入系统提示词

```typescript
const memoryContent = await loadMemories(botId)
const memorySection = memoryContent
  ? `## 用户长期记忆\n\n${memoryContent}`
  : ''
```

### 5.3 容量控制

memories.md 只存 active 记忆，被取代/删除的移到 archive.md。正常使用下几百条记忆，文件大小在几十 KB，注入 prompt 消耗几百 tokens，完全可控。

**硬上限：memories.md 超过 100 条 active 记忆时，只注入全部 P0 + 最近更新的 30 条 P1/P2。** 防止 prompt 超长导致隐性质量下降。

## 6. 生命周期管理

### 6.1 过期清理

```typescript
async function cleanExpiredMemories(botId: number): Promise<void> {
  const now = Date.now()
  await updateMemoryFile(botId, (content) => {
    const lines = content.split('\n')
    const toArchive: string[] = []

    const updated = lines.filter(line => {
      const match = line.match(/<!-- id:(\S+) .* updated:(\S+) -->/)
      if (!match) return true

      const priority = line.match(/\[(P[012])\]/)?.[1]
      const updatedAt = new Date(match[2]).getTime()
      const age = now - updatedAt

      const maxAge = priority === 'P1' ? 90 * DAY
                   : priority === 'P2' ? 30 * DAY
                   : Infinity

      if (age > maxAge) {
        toArchive.push(line)
        return false  // 从 memories.md 移除
      }
      return true
    })

    if (toArchive.length > 0) {
      appendToArchive(botId, toArchive)
    }
    return updated.join('\n')
  })
}
```

启动时或每日执行一次即可。

### 6.2 用户主动删除

通过提取 Prompt 的 `delete` action 自然处理，不需要额外机制。

## 7. 与 Qdrant 的关系

**MVP 阶段：Qdrant memories 集合直接停用。**

- 当前 Qdrant memories 只用了 BM25 稀疏向量，检索能力有限
- 百条级记忆全量加载完全够用，不需要向量检索
- 减少双写一致性的复杂度

**知识库（kb_*）不动。** RAG 检索仍走 Qdrant。

**后续如需语义检索**：JSONL 索引文件可以作为 Qdrant 同步的数据源，但 Markdown 始终是 source of truth。

## 9. 安全与合规

MVP 阶段最小处理：

- **敏感信息**：提取 Prompt 中明确约束不提取密码、密钥、token
- **删除权**：支持 delete action，用户说"忘掉 xxx"即可删除
- **访问隔离**：记忆按 botId 目录隔离

## 10. 验收标准

| 指标 | 目标 |
| ---- | ---- |
| 事实提取准确率 | 手动抽检 50 轮对话，有价值信息的召回率 > 80% |
| 误记率 | 提取出的条目中，无价值/错误的 < 10% |
| 去重有效性 | 同一事实不出现 > 2 条 active 记忆 |
| 性能 | 记忆加载 < 50ms（500 条以内），提取不阻塞响应流 |
| 事实变更 | 用户纠正旧事实后，旧记忆被原地更新为新内容 |
| targetId 容错 | targetId 不存在时，update 降级为 create，delete 静默跳过 |
