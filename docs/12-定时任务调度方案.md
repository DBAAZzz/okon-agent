# 定时任务调度方案

## 背景

当前 okon-agent 缺少主动调度能力，所有行为都是被动响应（用户发消息 → agent 回复）。随着功能演进，以下场景需要定时/延时执行：

- 记忆过期清理（当前是惰性触发，依赖用户对话才会执行）
- 定时摘要/日报生成
- 用户设置的提醒（"30分钟后提醒我..."）
- 知识库定期同步/索引刷新
- Channel 健康检查与自动重连

## 设计原则

1. **内嵌 Gateway，不引入外部依赖**（参考 OpenClaw 方案）
2. **任务持久化到文件系统**，进程重启后自动恢复
3. **轻量实现**，不过度设计，当前阶段不需要分布式调度
4. **与现有 memory / channel 体系自然融合**

## 整体架构

```
┌─────────────────────────────────────────┐
│                Gateway                   │
│                                          │
│  ┌──────────┐  ┌──────────────────────┐ │
│  │ Scheduler │──│  Job Store (JSON)    │ │
│  │  (tick)   │  │  data/scheduler/     │ │
│  └────┬─────┘  └──────────────────────┘ │
│       │                                  │
│       ├── CronExecutor (cron 表达式)     │
│       ├── IntervalExecutor (every N ms)  │
│       └── OnceExecutor (at timestamp)    │
│               │                          │
│               ▼                          │
│       ┌──────────────┐                   │
│       │  JobRunner    │                   │
│       │  - agent turn │                   │
│       │  - webhook    │                   │
│       │  - channel msg│                   │
│       └──────────────┘                   │
└─────────────────────────────────────────┘
```

## 核心数据结构

### Job 定义

```typescript
interface ScheduledJob {
  id: string                    // nanoid
  botId: number                 // 所属 bot
  name: string                  // 任务名称（人类可读）
  enabled: boolean

  // 调度方式（三选一）
  schedule:
    | { type: 'cron'; expression: string; timezone?: string }
    | { type: 'every'; intervalMs: number }
    | { type: 'at'; timestamp: number }

  // 执行方式
  action:
    | { type: 'agent-turn'; prompt: string; sessionId?: number }
    | { type: 'webhook'; url: string; method?: string; body?: unknown }
    | { type: 'channel-message'; channelType: string; chatId: string; message: string }
    | { type: 'internal'; handler: string }  // 内置任务，如 'memory:cleanup'

  // 执行配置
  config?: {
    model?: string              // 覆盖默认模型
    maxRetries?: number         // 失败重试次数，默认 0
    timeoutMs?: number          // 超时时间，默认 60s
    deleteAfterRun?: boolean    // 一次性任务执行后自动删除
  }

  // 元数据
  createdAt: string             // ISO 8601
  lastRunAt?: string
  lastRunStatus?: 'success' | 'failed'
  nextRunAt?: string
}
```

### 持久化存储

```
data/scheduler/
  └── jobs.json          # 所有 job 定义
  └── runs/              # 运行记录（可选，用于调试）
      └── {jobId}-{timestamp}.json
```

`jobs.json` 示例：

```json
[
  {
    "id": "abc123",
    "botId": 1,
    "name": "每日记忆清理",
    "enabled": true,
    "schedule": { "type": "cron", "expression": "0 3 * * *", "timezone": "Asia/Shanghai" },
    "action": { "type": "internal", "handler": "memory:cleanup" },
    "createdAt": "2026-02-26T10:00:00Z"
  },
  {
    "id": "def456",
    "botId": 1,
    "name": "用户提醒",
    "enabled": true,
    "schedule": { "type": "at", "timestamp": 1740600000000 },
    "action": { "type": "agent-turn", "prompt": "提醒用户：该开会了" },
    "config": { "deleteAfterRun": true },
    "createdAt": "2026-02-26T10:30:00Z"
  }
]
```

## 核心模块设计

### 1. Scheduler（调度器主体）

```typescript
// apps/agent/src/capabilities/scheduler/scheduler.ts

class Scheduler {
  private jobs: Map<string, ScheduledJob>
  private timers: Map<string, NodeJS.Timeout>
  private jobRunner: JobRunner

  /** 启动调度器，从文件加载 jobs 并注册定时器 */
  async start(): Promise<void>

  /** 停止所有定时器，优雅关闭 */
  async stop(): Promise<void>

  /** 添加任务（持久化 + 注册定时器） */
  async addJob(job: Omit<ScheduledJob, 'id' | 'createdAt'>): Promise<ScheduledJob>

  /** 删除任务 */
  async removeJob(jobId: string): Promise<void>

  /** 启用/禁用任务 */
  async toggleJob(jobId: string, enabled: boolean): Promise<void>

  /** 列出所有任务 */
  listJobs(botId?: number): ScheduledJob[]
}
```

核心 tick 逻辑：

- `cron` 类型：用 `cron-parser` 解析表达式，计算下次执行时间，`setTimeout` 到点触发
- `every` 类型：直接 `setInterval`
- `at` 类型：计算距离目标时间的 delay，`setTimeout` 一次性触发

每次执行完毕后：
1. 更新 `lastRunAt` 和 `lastRunStatus`
2. 计算并更新 `nextRunAt`
3. 如果 `deleteAfterRun`，自动移除
4. 持久化到 `jobs.json`

### 2. JobRunner（任务执行器）

```typescript
// apps/agent/src/capabilities/scheduler/job-runner.ts

class JobRunner {
  constructor(
    private gateway: Gateway,
    private channelManager: ChannelManager,
  ) {}

  async run(job: ScheduledJob): Promise<RunResult> {
    switch (job.action.type) {
      case 'agent-turn':
        // 创建或复用 session，调用 gateway.runAgent()
        // 相当于模拟一次用户消息触发 agent 执行
        return this.runAgentTurn(job)

      case 'channel-message':
        // 通过 channelManager 发送消息到指定渠道
        return this.sendChannelMessage(job)

      case 'webhook':
        // HTTP 请求
        return this.callWebhook(job)

      case 'internal':
        // 调用内置处理器（memory:cleanup 等）
        return this.runInternal(job)
    }
  }
}
```

### 3. JobStore（持久化层）

```typescript
// apps/agent/src/capabilities/scheduler/job-store.ts

class JobStore {
  private filePath: string  // data/scheduler/jobs.json

  async load(): Promise<ScheduledJob[]>
  async save(jobs: ScheduledJob[]): Promise<void>
  async saveRunRecord(jobId: string, result: RunResult): Promise<void>
}
```

写入策略与 memory 系统一致：先写 tmp 文件再 rename，保证原子性。

## 与现有系统的集成

### server.ts 启动流程变更

```typescript
// server.ts 中新增
import { createScheduler } from './capabilities/scheduler'

const scheduler = createScheduler({
  gateway,
  channelManager,
  memoryStore,
  dataDir: 'data/scheduler',
})

// 注册内置任务
scheduler.registerInternalHandler('memory:cleanup', async (job) => {
  await memoryStore.cleanExpired(job.botId)
})

// 服务启动后开始调度
fastify.addHook('onReady', async () => {
  await scheduler.start()
})

// 优雅关闭
fastify.addHook('onClose', async () => {
  await scheduler.stop()
})
```

### gateway.ts 集成

Agent 可以通过 tool calling 创建定时任务：

```typescript
// 新增 tool：schedule_task
{
  name: 'schedule_task',
  description: '创建定时任务或提醒',
  parameters: {
    name: { type: 'string', description: '任务名称' },
    schedule: { type: 'object', description: '调度配置' },
    action: { type: 'object', description: '执行动作' },
  },
  execute: async (params) => {
    const job = await scheduler.addJob({
      botId: currentBotId,
      ...params,
    })
    return `已创建定时任务：${job.name}（ID: ${job.id}）`
  },
}
```

这样用户可以自然语言创建提醒：

> 用户："每天早上9点给我发一条天气摘要"
> Agent 调用 schedule_task → cron: "0 9 * * *" + agent-turn: "查询天气并生成摘要"

## 分阶段实施

### Phase 1：基础调度（建议先做）

- Scheduler + JobStore + JobRunner 核心实现
- 支持 `internal` 类型任务
- 注册 `memory:cleanup` 内置任务
- server.ts 集成启动/关闭
- 依赖：`cron-parser`（纯解析库，无运行时依赖）

### Phase 2：Agent 集成

- 新增 `schedule_task` / `list_tasks` / `cancel_task` 工具
- 支持 `agent-turn` 类型任务
- 支持 `at` 一次性提醒

### Phase 3：Channel 投递

- 支持 `channel-message` 类型
- 定时任务结果推送到飞书等渠道
- 支持 `webhook` 类型

## 与 OpenClaw 方案的对比

| 维度 | OpenClaw | okon-agent（本方案） |
|------|----------|---------------------|
| 调度器位置 | Gateway 内嵌 | Gateway 内嵌（一致） |
| 持久化 | 磁盘 | JSON 文件（一致） |
| 调度模式 | cron / every / at | cron / every / at（一致） |
| 执行模式 | main session / isolated | agent-turn（类似 isolated） |
| 投递渠道 | 多渠道 + webhook | channel-message + webhook |
| 复杂度 | 较高（功能全面） | 精简（按需扩展） |

## 注意事项

1. **Gateway 必须常驻运行**：调度器在进程内，停机 = 停调度。生产环境建议用 PM2 / systemd 守护
2. **时钟漂移**：`at` 类型任务如果错过执行时间（比如重启期间），启动时应检查并立即补执行
3. **并发控制**：同一个 job 的上一次执行未完成时，跳过本次触发，避免堆积
4. **日志与可观测性**：RunRecord 保留最近 N 条，方便排查问题
